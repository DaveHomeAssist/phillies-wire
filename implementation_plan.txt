1. PROJECT SUMMARY

Objective
- Complete the merged Phillies dashboard by unifying the live Wire dashboard, innings view, schedule tracker, attendance state, and shared feeds into one repo owned Phillies command center inside `phillies-wire`.

Key outcomes
- `phillies-wire` becomes the canonical Phillies surface for dashboard, schedule, archive, ticker, and calendar outputs.
- A repo owned canonical schedule data layer is introduced and used by the dashboard, schedule view, JSON feeds, and ICS export.
- A new `/schedule/` surface ships inside `phillies-wire` with attendance tracking, notes, and clear links back to the main dashboard.
- `/dashboard/` and `/dashboard/innings/` read the same canonical game and schedule data as the schedule surface.
- External consumer outputs stay stable or gain versioned replacements, including `latest.json`, per issue `data.json`, ticker embed, archive JSON, and calendar feed.
- Phillies specific schedule ownership is removed from Ballparks Quest, leaving a clean handoff path or stub link.
- Verification, deploy, and smoke checks cover the merged system before cutover.

Assumptions
- `phillies-wire` remains a zero backend static site deployed by GitHub Pages through GitHub Actions.
- MLB Stats API remains the authoritative upstream for schedule and live game data.
- Manual schedule overrides are allowed, but they are file based, deterministic, and checked into the repo.
- Notion is planning and mirror context only, not a runtime data source.
- The current dashboard, innings view, `latest.json`, ticker embed, and `data.json` contract remain valid starting points and should be extended, not replaced.
- The canonical schedule store for this implementation is a repo owned JSON layer, not Postgres or SQLite, because the active repo contract is static and zero backend.
- Ballparks Quest changes are limited to migration, link cleanup, and Phillies schedule retirement, not a redesign of that product.

3. EXECUTION NOTES

Critical sequencing constraints
- Freeze the merged route and data contract before touching dashboard UI code.
- Build the canonical schedule layer before schedule UI, dashboard merge work, or ICS changes.
- Complete attendance state mapping before Ballparks Quest cutover.
- Keep `latest.json`, `archive.json`, and per issue `data.json` backward compatible until the merged dashboard is fully verified.
- Do not remove Phillies schedule ownership from Ballparks Quest until `/schedule/` is live in `phillies-wire` and migration checks pass.
- Verification updates must land in the same phase as the corresponding contract change, not at the end.

Parallelizable work
- Schedule schema design and Ballparks Quest route inventory can run in parallel during Phase 1.
- UI layout work for `/schedule/` can begin while the canonical schedule builder is being finalized, as long as the field map is frozen.
- Feed and ICS template work can run in parallel with dashboard merge work after the canonical schedule artifact is stable.
- Documentation updates can start during verification once route names and contracts stop moving.

Quick wins vs long lead tasks
- Quick wins:
- Freeze the route map and canonical field list.
- Add the canonical schedule artifact path and override file contract.
- Reuse the existing dashboard tokens, motion guards, and navigation shell for `/schedule/`.
- Add verification stubs for the new schedule artifact early.
- Long lead tasks:
- Attendance state migration from Ballparks Quest.
- Canonical schedule normalization for postponements and doubleheaders.
- Cross surface verification across dashboard, innings, schedule, ticker, and ICS.
- Ballparks Quest cutover and bookmark safe stubs.
