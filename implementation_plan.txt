  PHILLIES WIRE — crawl.mjs DECOMPOSITION PLAN
  Author: Claude Code review session (2026-04-15)
  Target: split the 1,400-line crawl.mjs monolith into focused modules
================================================================================

CONTEXT
-------
crawl.mjs is the heart of the Phillies Wire pipeline. It fetches MLB + weather
data, merges with the fixture baseline, and writes phillies-wire-data.json for
the enrich/render/verify stages to consume.

Current state: 1,404 lines in a single file, 46 functions, covering:
  - HTTP fetching (fetchJson, fetchSoft)
  - Payload building (buildLivePayload, buildOffDayPayload)
  - Section builders (buildHero, buildTicker, buildLineupSection, buildRotation,
    buildStandingsPreview, buildRecapPreview, buildSourceNotes, etc.)
  - Data extraction (extractKeyPerformers, extractDecisions, extractBattingOrder,
    resolvePitcher, resolveSeriesLabel)
  - Formatting helpers (formatGameTime, formatShortDate, formatWeekday, ...)
  - Validation (validateCrawlPayload)
  - Overrides + deep merge
  - Schema application

The file works. 33 unit tests + 1 integration smoke test cover it. The goal of
this refactor is NOT to change behavior but to make future changes safer and
reviews easier.

GOAL
----
Split crawl.mjs into a crawl/ subdirectory with one module per concern. After
the split:
  - crawl.mjs becomes a thin orchestrator (~150 lines)
  - Each extracted module has a focused responsibility and explicit exports
  - All 34 existing tests pass unchanged
  - Any test importing helpers from crawl.mjs updates its import path
  - Build artifact (phillies-wire-data.json) is byte-identical for a fixed input

ASSUMPTIONS
-----------
- Node 20+ with ESM ("type": "module" already set)
- Tests are the safety net. Every extraction must keep them green.
- The schema fixture (phillies-wire-schema.json) is representative enough to
  smoke-test the full pipeline after each phase.
- crawl.mjs is the only file importing these internals. Verified via grep:
  only test/crawl-helpers.test.mjs imports helpers directly.
- No consumers outside the repo depend on crawl.mjs's internal exports.

PROPOSED STRUCTURE
------------------
crawl/
  index.mjs                 entry point, orchestrates the pipeline
  api/
    mlb.mjs                 all MLB Stats API calls + url builders
    weather.mjs             Open-Meteo fetch
    http.mjs                fetchJson (with FETCH_TIMEOUT_MS), fetchSoft
  build/
    hero.mjs                buildHero + mode derivation
    ticker.mjs              buildTicker + daily prophet integration
    lineup.mjs              buildLineupSection, extractBattingOrder, buildTbdBattingOrder
    rotation.mjs            buildRotation, fetchPitcherStats
    recap.mjs               buildRecapPreview, buildRecapSummaryLine, extractKeyPerformers, extractDecisions
    roster.mjs              buildRosterHighlights, mergeInjuryEntries, normalizeLiveInjuries
    standings.mjs           fetchStandings, buildStandingsPreview, normalizeGamesBack
    game-status.mjs         build the game_status section incl. starters/series/broadcast/weather
    preview.mjs             buildUpNext + preview section wiring
    source-notes.mjs        buildSourceNotes
  resolve/
    pitcher.mjs             resolvePitcher
    series.mjs              resolveSeriesLabel
  payload/
    live.mjs                buildLivePayload (composes all section builders)
    off-day.mjs             buildOffDayPayload
    game-picker.mjs         pickActiveGame, collectUpcomingGames, isFinalGame
  validate.mjs              validateCrawlPayload, assertNoMojibake
  overrides.mjs             loadOverrides, applyOverrides, stripHeroOverride,
                            clampOverride, deepMerge (with __delete__ sentinel)
  format.mjs                formatGameTime, formatShortDate, formatMonthDay,
                            formatWeekday, formatGeneratedAtEt, buildWindSummary,
                            weatherCodeToText, getIsoDate, getRelativeIsoDate
  util.mjs                  cloneJson, dedupeStrings, shortTransactionNote,
                            inferPositionFromDescription

crawl.mjs (at repo root) remains as a compatibility shim that re-exports from
crawl/index.mjs so existing imports keep working until the test file is moved.

ASSUMPTIONS STATED:
- Tests currently import from ../crawl.mjs via scripts/lint.mjs and
  test/crawl-helpers.test.mjs. These imports must keep working until Phase 7.
- `resolvePitcher` and `resolveSeriesLabel` already live in their own helper
  form and are exported; other helpers need to be promoted to named exports.

PHASE 5 — EXTRACT build/ section builders (1-2 days)   MEDIUM-HIGH RISK
--------------------------------------------------------------------------------
This is the main structural work. Each section builder is a pure function that
takes a fixture + live data and returns a section object. Extract one at a time
and verify the snapshot after each.

Order (easiest first):
  1. crawl/build/source-notes.mjs     (no dependencies on other sections)
  2. crawl/build/ticker.mjs           (depends on resolvePitcher + format.mjs)
  3. crawl/build/roster.mjs           (injury merge, roster highlights)
  4. crawl/build/rotation.mjs         (depends on fetchPitcherStats, format)
  5. crawl/build/standings.mjs        (depends on fetchStandings, normalizeGamesBack)
  6. crawl/build/recap.mjs            (depends on resolvePitcher, extractDecisions, extractKeyPerformers)
  7. crawl/build/lineup.mjs           (depends on extractBattingOrder, buildTbdBattingOrder)
  8. crawl/build/game-status.mjs      (depends on many — hand this one last)
  9. crawl/build/preview.mjs          (depends on buildUpNext + pregame-preview.js)
  10. crawl/build/hero.mjs            (depends on everything — extract last)

For each extraction:
  - Create the new file with named exports
  - Update crawl.mjs import
  - Remove the original function from crawl.mjs
  - Run `npm test` and the snapshot oracle
  - Commit the single extraction

Milestones:
  - 10 focused commits, one per section
  - Each commit is individually revertable
  - After Phase 5, crawl.mjs no longer contains any function named `build*`

Resources:
  - 1 engineer
  - Snapshot oracle is the single source of truth — if it diverges, revert
    and investigate before continuing

Risk: side effects / shared state. Some builders mutate the data object in
place rather than returning a new section. These must keep doing exactly that
or the payload will drift. Snapshot catches it.

Exit criteria:
  - All section builders live under crawl/build/
  - Tests pass, snapshot matches
  - crawl.mjs is < 400 lines

RISK REGISTER
================================================================================

R1. Silent payload drift
    Likelihood: Medium. Some builders mutate the data object in place.
    Mitigation: snapshot oracle compares byte-for-byte after every extraction.

R2. Import cycle between build modules
    Likelihood: Low. Dependencies flow one way (util -> format -> api ->
    resolve -> build -> payload).
    Mitigation: strict layer ordering. If a build module needs another build
    module, the shared logic belongs in util.mjs or resolve/.

R3. Production cron runs during the refactor
    Likelihood: Medium. The 15-min game-window cron fires frequently.
    Mitigation: do the refactor on a feature branch, never push to main mid-
    phase. Schedule the final PR merge during a no-game window.

R4. Hydrate parameter ordering in URL changes
    Likelihood: Low but high impact.
    Mitigation: snapshot the exact URL strings in the http layer. Freeze them
    as string constants, not template-concatenation.

R5. Test file import paths break
    Likelihood: Medium. test/crawl-helpers.test.mjs imports from ../crawl.mjs.
    Mitigation: keep crawl.mjs as a compatibility shim until Phase 7.
